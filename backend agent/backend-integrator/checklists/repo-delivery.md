# Repo Delivery Checklist

Use this checklist to validate that a repository structure generated by the Repo Builder Agent is complete, coherent, and production-ready.

Run this checklist after completing a `/scaffold` or `/assemble` output, and before handing the repo to engineers.

---

## 1. Repository Structure

- [ ] All applications live inside `apps/` directory.
- [ ] Reusable code lives inside `packages/` directory.
- [ ] Infrastructure files live inside `infra/` directory.
- [ ] Documentation lives inside `docs/` or root-level markdown files.
- [ ] No application code is mixed inside infrastructure directories.
- [ ] No infrastructure files are mixed inside application directories.

---

## 2. Application Separation

- [ ] Frontend application is correctly isolated (e.g., `apps/web`).
- [ ] Backend API is correctly isolated (e.g., `apps/api`).
- [ ] Background worker is isolated (e.g., `apps/worker`) — if background jobs exist.
- [ ] No shared runtime code exists directly between services (only via packages).
- [ ] Each app has its own `package.json` with its own scripts.

---

## 3. Shared Packages

- [ ] Shared TypeScript types live in `packages/types`.
- [ ] Internal API client (SDK) lives in `packages/sdk` — if needed.
- [ ] Shared configuration lives in `packages/config` — if needed.
- [ ] Each package has a clear, documented purpose.
- [ ] No package exists without at least one consumer service.
- [ ] Package dependency graph is acyclic (no circular dependencies).

---

## 4. Environment Configuration

- [ ] `.env.example` exists at the root (or per-service for multi-app repos).
- [ ] All required environment variables are documented in `.env.example`.
- [ ] Variables are grouped logically (Database, Auth, Integrations, etc.).
- [ ] No default secrets or real credentials exist in `.env.example`.
- [ ] `.gitignore` covers `.env`, `.env.local`, `.env.*` (excluding `.example`).

---

## 5. Infrastructure

- [ ] `docker-compose.yml` exists for local development.
- [ ] All required services are defined in `docker-compose.yml` (database, cache, queue, etc.).
- [ ] Each backend service has a `Dockerfile`.
- [ ] Dockerfiles do not run as root.
- [ ] Database migration workflow is defined (location + command documented).
- [ ] Database seed workflow is defined (if applicable).
- [ ] Local development can be started with a single command.

---

## 6. CI/CD Skeleton

- [ ] `.github/workflows/ci.yml` (or equivalent) exists.
- [ ] CI pipeline installs dependencies.
- [ ] CI pipeline runs lint checks.
- [ ] CI pipeline runs tests.
- [ ] CI pipeline runs build.
- [ ] CI pipeline includes a security/dependency scan step.

---

## 7. Security Baseline

- [ ] `SECURITY.md` exists with vulnerability reporting instructions.
- [ ] `.gitleaks.toml` or equivalent secret scanning config exists.
- [ ] `.gitignore` is comprehensive for the project stack.
- [ ] CI security scan step is present (e.g., `npm audit`, `trivy`).

---

## 8. Documentation

- [ ] `README.md` exists and explains the repository structure.
- [ ] `SETUP.md` exists and provides a step-by-step local setup guide.
- [ ] `ARCHITECTURE.md` exists and explains how the system is organized.
- [ ] `ARCHITECTURE.md` contains a Mermaid dependency diagram (Rule 23).
- [ ] Each `apps/*` directory has a minimal README.
- [ ] Environment variable requirements are documented per service.

---

## 9. Developer Experience

- [ ] A developer can clone → install → configure → run in under 5 minutes.
- [ ] Root `package.json` has a `dev` script that starts all services.
- [ ] Root `package.json` has scripts for: `install`, `build`, `test`, `lint`.
- [ ] Migration and seed scripts are runnable with documented commands.

---

## 10. Handoff Artifact

- [ ] `repo-handoff.json` has been generated.
- [ ] `repo-handoff.json` is valid against `repo-handoff.schema.json`.
- [ ] `meta.generatedAt` is populated with the correct timestamp.
- [ ] `meta.modeUsed` reflects the command that was used.
- [ ] All assumptions made by the agent are documented in `assumptions[]`.

---

## Scoring

A repository is **not ready for engineer handoff** if:

- Any item in sections 1, 4, or 7 is unchecked.
- More than 2 items in sections 5 or 8 are unchecked.
- Section 10 (Handoff Artifact) is incomplete.

Address all gaps before delivering the repository to the engineering team.
